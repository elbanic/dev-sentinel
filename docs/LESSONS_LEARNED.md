# Lessons Learned from claude-sentinel v1

> Technical lessons extracted from building claude-sentinel. These are real failures — not hypothetical.

---

## Architecture Lessons

### 1. Dual-hook sentiment coupling was the root complexity

**What happened**: UserPromptSubmit stored sentiment per turn to SQLite. Stop hook read those turns back, replayed them through a ConversationObserver state machine, and detected patterns. This created tight coupling between hooks via the DB.

**Why it was wrong**: Two hooks sharing state through SQLite meant:
- Stop hook couldn't work without UserPromptSubmit having stored data
- Observer replay logic duplicated sentiment understanding
- Debugging required tracing data across both hooks + DB

**Lesson**: Frustration detection should happen once, at a single point. Flag the result, don't replay it.

### 2. ConversationObserver state machine was unnecessary

**What happened**: Built a state machine (normal → struggling → resolved → abandoned) that replayed stored turns to reconstruct conversation state.

**Why it was wrong**: The state machine was a middleman between "LLM says frustrated" and "generate draft". It added complexity without adding value — the LLM already knows if the user is frustrated.

**Lesson**: If the LLM can answer the question directly, don't build a state machine to approximate the answer.

### 3. Over-abstracted Agentic RAG pipeline

**What happened**: Built CachedIntentAnalyzer, ParallelMemorySearchAgent, StreamingAgenticPipeline, ReflectiveJudgeAgent as wrapper layers around simple operations.

**Why it was wrong**: Each wrapper added indirection without meaningful behavior change. The "parallel" wrapper just called the inner agent. The "cached" wrapper added LRU cache overhead for operations that ran once per prompt. The "reflective" wrapper called the judge twice.

**Lesson**: Start with direct function calls. Add caching/parallelism/streaming only when profiling shows a real bottleneck.

## LLM Integration Lessons

### 4. Keyword-based fallback violated LLM-only principle

**What happened**: When LLM was unavailable, fell back to keyword matching for intent classification. `proposedApproach` was generated by `keywords.join(', ')`, producing garbage like `"path, not there, looked, config, not working, why"`.

**Lesson**: Either use LLM or skip the analysis. Half-measures produce worse results than no results. Graceful degradation means "skip", not "use keywords".

### 5. `summarizeFailure()` returned plain text but caller expected JSON

**What happened**: `note-generator.ts` called `llmProvider.summarizeFailure()` and then tried `JSON.parse()` on the result. `summarizeFailure()` always returned a plain text paragraph, so JSON parse always failed, and LLM lesson extraction never worked.

**Root cause**: `LLMProvider` interface lacked `generateCompletion(systemPrompt, userPrompt)`. Each method had its own hardcoded prompt, so callers couldn't use the purpose-built `PROMPTS.lessonSummarization` prompt.

**Fix applied**: Added `generateCompletion()` to the interface. All providers already had it as `private` — just made it `public`.

**Lesson**: LLM provider should expose a generic `generateCompletion(system, user)` method. Don't force callers to use purpose-specific methods with hardcoded prompts.

### 6. Ollama sequential processing causes test flakiness

**What happened**: Integration tests that spawn real CLI processes connecting to Ollama intermittently fail with timeouts. Ollama processes requests sequentially — when two LLM calls run in "parallel" (Promise.all), the second one queues behind the first.

**Details**:
- qwen3:4b is a reasoning model that generates `<think>` blocks (~600 tokens) before each response → ~20s per call
- Two "parallel" calls (intent + sentiment) actually take ~40s sequentially through Ollama
- Pipeline timeout (15s) regularly expires before both complete

**Lesson**:
- All unit/property tests should use MockLLMProvider. Only integration tests should touch Ollama.
- Consider `OLLAMA_NUM_PARALLEL=2` env var to enable actual parallel processing
- Consider adding `num_predict` limit to reduce thinking token overhead
- Tests depending on external LLM should be tagged and run separately

## Data & Storage Lessons

### 7. Vector store needs dimension migration support

**What happened**: Changed embedding model from `nomic-embed-text` (768 dims) to `qwen3-embedding` (1024 dims). Existing vectors in the DB became incompatible.

**Lesson**: Vector store should have `clearVectors()` for re-indexing, and store the model name as metadata.

### 8. `sentinel review confirm` didn't write to vector store

**What happened**: Confirming a draft wrote to the `experiences` SQLite table but not to the vector store. Semantic search always returned empty because there were no vectors.

**Lesson**: The confirm path must write to BOTH stores (SQLite metadata + vector embeddings) atomically.

## Hook Protocol Lessons

### 9. Stop hook cannot return systemMessage

**What happened**: Tried to notify the user about detected patterns from the Stop hook.

**Reality**: Stop hook protocol only supports `{ "decision": "approve" }`. No systemMessage field.

**Workaround**: Save draft in Stop hook, notify user in the NEXT UserPromptSubmit via systemMessage.

### 10. Each hook invocation is a new process

**What happened**: Assumed in-memory state would persist between hook calls.

**Reality**: Claude Code spawns a new process for each hook invocation. No in-memory state survives.

**Lesson**: All state must go through SQLite. Design for stateless invocations.

### 11. Stop hook fires on EVERY Claude response, not just session end

**What happened**: Assumed Stop hook fires once at session end.

**Reality**: It fires after every Claude response. Need dedup to avoid generating multiple drafts for the same session.

## Testing Lessons

### 12. Property-based tests caught real bugs

**What happened**: fast-check found edge cases that unit tests missed:
- Empty string prompts causing crashes
- Unicode (Korean, Japanese, Chinese) characters in technology extraction
- Very long prompts (>10KB) causing timeouts
- Concurrent calls producing inconsistent results

**Lesson**: Use fast-check for any function that takes string input. Arbitraries for Korean prompts, error patterns, and edge cases are reusable.

### 13. Mock LLM provider is critical for test determinism

**What happened**: Tests using real Ollama were flaky (timing-dependent, model-dependent).

**Lesson**: MockLLMProvider should be the default for ALL tests. It should:
- Return deterministic responses based on input patterns
- Support `shouldFail` mode for error path testing
- Implement the full `LLMProvider` interface (including `generateCompletion`)
- Track calls for assertion (spy functionality)

---

## Process Lessons

### 14. TDD prevented over-engineering

**What happened**: Writing tests first forced thinking about what the code should DO before how to implement it. When tests were clear, implementation was straightforward.

**Lesson**: test-architect → implementer → code-reviewer → refactorer loop works. MAX 3 iterations per cycle.

### 15. CLAUDE.md anti-patterns prevent regression

**What happened**: After removing keyword matching, a new feature accidentally reintroduced it.

**Lesson**: Document anti-patterns in CLAUDE.md. AI agents follow these rules if they're explicit.
