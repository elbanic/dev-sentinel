# Dev Sentinel v2 — Completing the Philosophy

> **What broke you guards you.**
>
> v1 turns failure into warnings. v2 turns warnings into growth.

In v1, experiences are static snapshots — captured once, recalled when similar frustration appears. v2 makes the entire system **alive**: experiences evolve, advice self-corrects, and the developer sees their growth over time.

---

## Feature 1: Experience Evolution

### Problem

A confirmed experience is immutable. But real knowledge isn't — the second time you solve a problem, you understand it deeper than the first. Currently, Sentinel treats the first solution as the final answer.

### Concept

When a developer encounters a problem that matches an existing experience and resolves it with a **better approach**, the experience evolves. The old solution demotes to `failedApproaches`, and the new insight takes its place.

```
Experience #1 (v1):
  frustration: "Jest mock leaking between tests"
  failed: ["Cleared Jest cache"]
  success: "Used jest.isolateModules()"
  lessons: ["Check module-level singletons"]

── 3 weeks later, same problem recurs ──

Experience #1 (v2):
  frustration: "Jest mock leaking between tests"
  failed: ["Cleared Jest cache", "Used jest.isolateModules() — worked but fragile"]
  success: "Refactored to dependency injection, eliminating module-level singleton"
  lessons: ["Check module-level singletons",
            "isolateModules is a bandaid — DI removes the root cause"]
  revisions: 2
```

### Flow

```
UserPromptSubmit hook (frustrated prompt detected)
  │
  ├─ searchMemory() finds matching experience
  │   └─ systemMessage: warning with past experience
  │
  ├─ ... developer works through the problem ...
  │
  ├─ resolution detected → flag upgraded to 'capture'
  │
  └─ Stop hook fires
      │
      ├─ Existing flow: store raw transcript as draft
      │
      └─ NEW: tag draft with matchedExperienceId
           (the experience that was warned about)

sentinel review confirm <id>
  │
  ├─ LLM summarization (existing)
  │
  └─ NEW: if matchedExperienceId exists
      │
      ├─ LLM comparison: "Is this a better solution
      │   than the previous one?"
      │
      ├─ YES → evolve existing experience
      │   ├─ old successfulApproach → failedApproaches
      │   ├─ new successfulApproach replaces it
      │   ├─ merge lessons (deduplicated)
      │   ├─ increment revision count
      │   └─ re-embed with updated text
      │
      └─ NO → store as separate experience
          (different problem, just happened to match)
```

### Example Scenario

```
Session 1:
  Developer: "TypeScript 빌드가 자꾸 실패해. path alias가 안 먹혀"
  Sentinel: (no prior experience) → captures draft
  Confirm: experience stored
    success: "tsconfig.json에 paths 추가 + jest.config에 moduleNameMapper 설정"

Session 2 (weeks later):
  Developer: "또 path alias 문제야. 새 패키지 추가할 때마다 두 군데 수정해야 해"
  Sentinel: "Past experience: tsconfig paths + moduleNameMapper로 해결한 적 있음"
  Developer: resolves with tsconfig-paths 패키지 도입 (single source of truth)
  Confirm: LLM judges this is a better solution
    → experience evolves
    failed: [..., "tsconfig paths + moduleNameMapper — works but duplicated config"]
    success: "tsconfig-paths 패키지로 tsconfig.json을 single source of truth로 통합"
    lessons: [..., "path alias 설정이 여러 곳에 분산되면 drift 발생. 단일 소스 유지"]
    revisions: 2
```

## Feature 2: Feedback Loop

### Problem

Sentinel gives advice, but never learns whether that advice actually helped. A warning that doesn't prevent failure is noise. A warning that consistently helps is validated knowledge.

### Concept

Track the **outcome** of each advice given. If the developer sailed through after a warning, the advice was effective. If they got frustrated again with the same problem, the advice was insufficient — the experience description or suggested action may need improvement.

```
Advice Outcome Tracking:

  advice given ──┬── session ends without re-frustration ── EFFECTIVE
                 │
                 └── same frustration recurs in session ──── INEFFECTIVE
                      │
                      └── experience flagged for review
                           "This advice didn't prevent the failure.
                            Consider enriching the experience."
```

### Flow

```
UserPromptSubmit hook (frustrated prompt detected)
  │
  ├─ searchMemory() → match found → advice given
  │   └─ recordAdvice(sessionId, experienceId)  ← existing
  │
  ├─ ... developer continues working ...
  │
  └─ LATER in same session, another frustrated prompt
      │
      ├─ analyzeFrustration() → frustrated again
      │
      └─ NEW: check if advice was given earlier in this session
          │
          ├─ LLM judge: "Is this frustration related to
          │   the experience we warned about?"
          │
          ├─ YES (same topic) → mark advice as INEFFECTIVE
          │   └─ record outcome: { experienceId, effective: false }
          │
          └─ NO (different topic) → normal flow
               (new frustration, unrelated to prior advice)

Session ends without re-frustration on advised topic:
  └─ Stop hook or next session start
      └─ mark advice as EFFECTIVE
         └─ record outcome: { experienceId, effective: true }
```

### Effectiveness Score

Each experience accumulates an effectiveness score over time:

```
Experience: "Docker build cache invalidation"
  Times advised: 8
  Effective: 6  (developer didn't re-frustrate)
  Ineffective: 2  (developer got stuck again despite warning)
  Effectiveness: 75%

Experience: "ESLint config inheritance in monorepo"
  Times advised: 3
  Effective: 1
  Ineffective: 2
  Effectiveness: 33%  ← flag for review
```

Low-effectiveness experiences surface in the dashboard as candidates for enrichment — their descriptions may be too vague, their lessons too shallow, or their solutions outdated.

### Example Scenario

```
Session 1:
  Developer: "Docker 빌드가 캐시를 안 타고 매번 처음부터 해"
  Sentinel: "Past experience: COPY package*.json 을 먼저 하고
             npm install 후에 COPY . 하면 캐시 활용 가능"
  Developer: follows advice, build works fast
  → outcome: EFFECTIVE

Session 2:
  Developer: "또 Docker 빌드 캐시가 안 먹혀"
  Sentinel: same advice given
  Developer: "이미 그렇게 했는데도 안 돼. .dockerignore에 node_modules 빠져있었음"
  → outcome: INEFFECTIVE
  → experience flagged: "advice didn't cover .dockerignore case"
  → candidate for enrichment: add lesson about .dockerignore
```

## Feature 3: Sentinel Dashboard

### Problem

Sentinel's data lives in SQLite, accessible only through CLI text output. The developer has no visual sense of their growth — how many failures they've turned into knowledge, whether their patterns are improving, or which areas need attention.

### Concept

A local web dashboard that visualizes the developer's failure-to-knowledge journey. Includes session retrospective (on-demand review of any session), experience health metrics, and growth tracking over time.

### Components

```
sentinel dashboard
  │
  ├─ Overview
  │   ├─ Total experiences (with growth trend)
  │   ├─ Pending drafts count
  │   ├─ Advice effectiveness rate (from Feedback Loop)
  │   └─ Active streak (days since last unresolved frustration)
  │
  ├─ Experience Explorer
  │   ├─ List all experiences (sortable by date, effectiveness, revision count)
  │   ├─ Detail view: full experience with evolution history
  │   ├─ Health indicators:
  │   │   ├─ green: high effectiveness, recently validated
  │   │   ├─ yellow: low effectiveness, needs enrichment
  │   │   └─ gray: never triggered, possibly outdated
  │   └─ Edit/enrich experience inline
  │
  ├─ Session Retrospective
  │   ├─ List recent sessions with frustration indicators
  │   ├─ On-demand LLM analysis: "What happened in this session?"
  │   ├─ Generate draft from any past session (not just frustrated ones)
  │   └─ Timeline view: prompt → frustration → advice → resolution
  │
  └─ Pattern Analysis (on-demand LLM batch analysis)
      ├─ Sentinel Insight: LLM-generated natural language summary
      ├─ Weak Areas: discovered categories with counts (bar chart)
      ├─ Frustration Trend: daily aggregation from session_turns
      ├─ Resolution Rate: resolved vs abandoned ratio
      ├─ Analysis caching: English base in DB, translate on language switch
      └─ Deep think when experiences >= 5
```

### Session Retrospective Detail

But valuable lessons exist in non-frustrated sessions too: long debugging sessions that eventually succeeded, sessions where the developer changed approach mid-way, or sessions where they learned something new without explicit frustration.

### Dashboard Architecture (TBD)

The dashboard requires separate design discussion. Key decisions:

- **Serving**: Local HTTP server (`sentinel dashboard` starts it) vs. static HTML generation
- **Frontend**: Minimal framework (vanilla JS + HTML) vs. lightweight library
- **Data flow**: Direct SQLite reads vs. REST API layer
- **Real-time**: Poll-based refresh vs. file watcher for live updates

These choices should follow the same philosophy — local-first, minimal abstraction, graceful degradation. The dashboard should work even if some data is missing or corrupted, displaying what it can rather than crashing.

## Implementation Priority

```
Phase 1: Experience Evolution
  └─ Foundation for living knowledge
     (extends existing confirm flow, minimal new infrastructure)

Phase 2: Feedback Loop
  └─ Self-correcting advice system
     (extends existing session_advices table, adds outcome tracking)

Phase 3: Sentinel Dashboard
  └─ Visualization layer on top of Phase 1 + 2 data
     (new infrastructure: web server, frontend, API)
```

Each phase independently delivers value. Phase 3 is most impactful when Phase 1 and 2 data is already flowing.
