# V2.1 Design: Capture Pipeline Redesign

## Problem Statement

Current capture pipeline requires the user to explicitly express resolution ("I solved it") or abandonment ("I give up") for the LLM to classify and trigger capture. In reality, developers almost never do this. When a problem is solved, they silently move on to the next task.

This means the `frustrated → resolution → capture` path rarely triggers in production. Most frustrated sessions end without being captured.

## Claude Code Session Lifecycle (from official docs)

### Key Findings

| Behavior | Detail |
|----------|--------|
| **Compact does NOT change session_id** | Same session continues after compact |
| **Transcript file persists after compact** | Original messages preserved in the transcript |
| **SessionEnd hook exists** | Fires when session truly ends, with `reason` field |
| **PreCompact hook exists** | Fires before compact, with `trigger` (manual/auto) |

### Available Hook Events (relevant to capture)

```
UserPromptSubmit  — every user prompt
Stop              — after every Claude response
PreCompact        — before context compaction
SessionEnd        — when session ends
```

## New Design: Dual-Trigger Capture

### Principle

Two capture triggers, whichever fires first ends the cycle:

1. **Explicit trigger**: User expresses resolution/abandonment (existing LLM classification)
2. **Implicit trigger**: Session ends while frustrated flag is active (new SessionEnd hook)

Both are valid. The explicit path captures with higher confidence about outcome; the implicit path ensures no frustrated session is lost.

### Flow Diagram

```
UserPromptSubmit:
  prompt → LLM frustration analysis
  ├─ frustrated → setFlag('frustrated') + searchMemory (Active Recall)
  ├─ resolution/abandonment → upgradeFlag('capture') ← Trigger 1 (explicit)
  └─ normal → pass through

Stop hook:
  flag === 'capture' → store transcript as draft → clearFlag
  (unchanged from current)

SessionEnd hook (NEW):
  flag === 'frustrated' → upgradeFlag('capture')  ← Trigger 2 (implicit)
                        → store transcript as draft
                        → clearFlag

User (async):
  sentinel review confirm → LLM judges resolution status + summarizes
```

### Trigger Comparison

| | Trigger 1: Explicit | Trigger 2: SessionEnd |
|---|---|---|
| **When** | User says "solved" or "giving up" | Session ends (close terminal, `/exit`, etc.) |
| **Confidence** | High — user stated outcome | Lower — outcome unknown |
| **Frequency** | Rare (users rarely announce) | Always fires if session ends |
| **Transcript** | Partial (up to resolution point) | Full session transcript |
| **Resolution known?** | Yes (resolution vs abandonment) | No — deferred to `review confirm` |

### SessionEnd Handler Implementation

```typescript
// New hook handler: src/hook/session-end-handler.ts

async function handleSessionEnd(input: {
  sessionId: string;
  transcriptPath: string;
  sqliteStore: SqliteStore;
}): Promise<void> {
  const { sessionId, transcriptPath, sqliteStore } = input;

  const flag = sqliteStore.getFlag(sessionId);
  if (!flag || flag.status !== 'frustrated') return;

  // Frustrated session ending — auto-capture
  sqliteStore.upgradeFlag(sessionId, 'capture');

  // Reuse existing capture pipeline
  runCapturePipeline(sessionId, transcriptPath, sqliteStore, flag.matched_experience_id ?? undefined);

  sqliteStore.clearFlag(sessionId);
}
```

### Review Confirm: Resolution Judgment

Since SessionEnd-triggered drafts don't know if the problem was resolved, the `review confirm` step needs to handle this:

```
sentinel review confirm <id>
  → LLM receives full transcript
  → Existing: extract frustrationSignature, failedApproaches, successfulApproach, lessons
  → successfulApproach may be null if problem was not resolved
  → If matchedExperienceId exists:
    → Evolution judge decides if new encounter is better
    → isBetter=true with no successfulApproach → still valuable (adds new failed approaches)
  → If no matchedExperienceId:
    → Store as new experience (successfulApproach may be null — that's OK)
```

Experiences without `successfulApproach` are still valuable — they document what NOT to do.

### Transcript Completeness

Since compact preserves original messages in the transcript file and does not change session_id:
- SessionEnd transcript contains the full session history including pre-compact messages
- No special handling needed for compact — the transcript file already has everything

If future Claude Code versions change this behavior, we may need to use the PreCompact hook to snapshot transcripts before compaction. For now, this is not necessary.

## Migration Plan

### What Changes

| Component | Change |
|-----------|--------|
| `src/hook/session-end-handler.ts` | New file — SessionEnd hook handler |
| `src/cli.ts` | Add `--hook session-end` routing |
| `src/cli/init-command.ts` | Register SessionEnd hook |
| `src/hook/user-prompt-submit-handler.ts` | Keep resolution/abandonment detection (no change) |
| `src/hook/stop-hook-handler.ts` | No change |
| `confirmExperience` | Allow `successfulApproach: null` for unresolved experiences |

### What Does NOT Change

- Frustration detection (UserPromptSubmit) — unchanged
- Active Recall (memory matching) — unchanged
- Stop hook capture pipeline — unchanged
- Evolution logic (review confirm) — unchanged
- Resolution/abandonment classification — **kept** as explicit trigger

### Backward Compatibility

- SessionEnd hook is additive — existing installations without it continue to work via explicit triggers
- `sentinel init` would add the new hook alongside existing ones
- Old drafts without SessionEnd are unaffected

## Open Questions

1. **SessionEnd `reason` field**: What values can it have? ("other" is shown in docs.) Could be useful for filtering — e.g., don't capture if reason is "error".
2. **Multiple frustrated episodes per session**: If a user gets frustrated, resolves it, gets frustrated again — the flag lifecycle handles this (clearFlag after capture, new setFlag on next frustration). But SessionEnd would only capture the last state.
3. **Very short sessions**: User opens Claude Code, types one frustrated prompt, closes immediately. SessionEnd fires, but transcript has minimal content. Should we set a minimum message count threshold?
